# KDD Init

Knowledge Driven Development（知识库驱动开发）初始化工具。让 AI 编码助手基于项目知识库工作，而不是基于猜测。

## 什么是知识库

整个项目就是知识库。代码表达了**能做什么、怎么做**；文档补充代码无法表达的内容——**禁止做什么、不可违反什么、踩过什么坑**。KDD 初始化的正是这部分文档。

## 为什么需要 KDD

AI 编码助手在没有项目上下文的情况下，容易：

- 猜测业务规则，写出不符合需求的代码
- 无视现有架构，随意引入新模式
- 违反业务不变量，绕过关键约束
- 重复踩坑，不吸取项目历史教训
- 修改代码后不更新文档，知识逐渐失真

KDD 通过建立项目知识库，让 AI 在每次任务前先阅读知识、理解约束、分析影响，再动手编码。

## 核心理念

```text
知识库聚焦代码无法表达的内容
约束、不变量、边界和术语
设计先于编码
文档先于实现
代码与知识同步
```

知识库不重复代码已表达的内容（数据结构、API 定义、表结构），只记录代码表达不了的内容（禁止事项、业务不变量、架构边界、术语定义）。

## 设计原则

KDD 遵循 6 条设计原则：

1. **业务优先** — 从业务能力出发设计，而非从数据库、框架、技术栈出发
2. **职责优先** — 每个模块拥有明确职责，高内聚低耦合
3. **三层隔离** — 策略层（易变）→ 核心层（稳定）← 技术层（可替换）。逻辑分层，业务内聚、变更隔离、依赖单向
4. **避免过度设计** — Simple First, Evolve Later
5. **变更成本** — 为已知变化设计，不为未知变化设计
6. **按变化演进** — 解决问题 → 发现变化 → 抽象能力 → 沉淀扩展点 → 持续演进

原则冲突时按优先级裁决：业务优先 > 职责优先 > 变更成本 > 简单优先 > 扩展优先

## 知识库内容

### 核心文档

所有项目都需要，初始化时自动生成。

| 文档 | 内容 | 为什么 AI 需要 |
|------|------|---------------|
| constraints.md | 架构约束、业务不变量、禁止事项、设计原则 | AI 从代码推断不出"禁止做的事"和"不可违反的规则" |
| glossary.md | 术语定义、缩写表 | 统一语言，防止 AI 用错概念 |

### 条件文档

按项目需要创建。

| 文档 | 创建条件 | 为什么 AI 需要 |
|------|---------|---------------|
| boundaries.md | 系统与外部有复杂交互 | AI 容易越界实现不属于系统的能力 |
| flows.md | 存在多步骤业务流程或复杂状态机 | AI 容易遗漏禁止的状态转换 |

### 积累文档

初始化时创建空目录，随项目演进持续积累。

| 文档 | 用途 |
|------|------|
| adr/ | 架构决策记录，做出重要技术决策时新增 |
| lessons/ | 经验教训库，踩坑/复盘时新增 |

### 为什么不记录数据结构、API 定义、表结构？

因为这些内容代码已经表达了，AI 可以直接读代码。文档重复记录这些内容只会增加维护负担，一旦过时比没有文档更危险。

知识库只记录代码的"负空间"——那些代码里看不到但必须遵守的规则。

### 为什么不需要详细业务文档？

因为 KDD 要求遵循三层隔离架构。三层隔离使核心业务规则内聚在核心层，策略决策内聚在策略层，技术实现内聚在技术层。AI 可以通过代码结构精准定位业务逻辑，无需额外文档描述"哪个模块负责什么"。

精简文档不是人为约束，而是三层隔离架构的自然结果。如果核心层不够内聚（业务规则散落在技术层或策略层），应优先修复代码结构，而非用文档补。

### 老项目没有三层隔离怎么办？

KDD 根据架构状态自适应文档策略，不需要用户手动选择方案：

| 架构状态 | AI 代码定位能力 | 文档策略 |
|----------|----------------|----------|
| 已实施 | 精准 | 只记录负空间 |
| 部分实施 | 部分精准 | 未隔离模块补充定位信息 |
| 未实施 | 模糊 | 补充业务规则定位信息 |

在 `constraints.md` 中标注架构状态，AI 会据此自动调整文档依赖程度。随着代码逐步重构，架构状态升级，文档逐步精简。

## 工作流程

所有任务必须经过七步门禁：

```text
知识库阅读 → 影响分析 → 设计方案 → 用户确认 → 开发实现 → 测试验证 → 知识库同步
```

纯 UI 调整可走轻量门禁，走简化流程。

## 快速开始

### 1. 安装 Skill

将本项目作为 Skill 安装到你的 AI 编码工具中。

### 2. 执行初始化

运行 `kdd-init` Skill，按提示操作：

1. 输入知识库父目录路径
2. 选择你使用的 Agent（Trae / Cursor / Claude Code / GitHub Copilot / Custom）
3. 加载对应 Agent 适配器
4. AI 自动分析代码，生成知识库文档
5. 完成初始化

### 3. 开始开发

初始化完成后，AI 会自动遵循 KDD 工作流：每次任务前先读知识库，分析影响，设计方案，获得确认后再编码。

## 项目结构

```text
kdd-init/
├── SKILL.md                          # Skill 入口
├── adapters/                         # 多 Agent 适配
│   ├── trae.md                       #   Trae
│   ├── cursor.md                     #   Cursor
│   ├── claude.md                     #   Claude Code
│   ├── copilot.md                    #   GitHub Copilot
│   └── custom.md                     #   自定义
└── templates/
    ├── standards/                    # AI 规则文件
    │   └── kdd-workflow.md           #   七步门禁工作流
    └── knowledge-base/               # 知识库模板
        ├── README.md                 #   知识库导航
        ├── constraints.md            #   约束（核心，必读）
        ├── glossary.md               #   术语表（核心）
        ├── boundaries.md             #   能力边界（条件）
        ├── flows.md                  #   业务流程与状态机（条件）
        ├── adr/                      #   架构决策记录（积累）
        └── lessons/                  #   经验教训库（积累）
```

## 支持的 Agent

| Agent | 规则目录 | 说明 |
|-------|---------|------|
| Trae | `.trae/rules/` | 直接复制 |
| Cursor | `.cursor/rules/` | 后缀改为 .mdc |
| Claude Code | `CLAUDE.md` | 合并为单文件 |
| GitHub Copilot | `.github/copilot-instructions.md` | 重命名文件 |
| Custom | 用户指定 | 直接复制 |
