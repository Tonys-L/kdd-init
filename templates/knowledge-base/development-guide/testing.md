# 测试规范

## 测试分层

| 测试类型 | 范围 | 运行频率 | 速度要求 |
|----------|------|----------|----------|
| 单元测试 | 单个函数/类 | 每次提交 | < 10ms/个 |
| 集成测试 | 模块间协作 | 每次提交 | < 1s/个 |
| E2E 测试 | 完整业务流程 | 合并前 | 无限制 |

---

## 测试文件位置

- 测试文件与源码同目录：`user-service.ts` → `user-service.test.ts`
- E2E 测试放在独立目录：`e2e/`

---

## 测试命名

```typescript
describe('UserService', () => {
  describe('createUser', () => {
    it('should create user with valid input')
    it('should throw when email is duplicated')
    it('should hash password before saving')
  })
})
```

- `describe` 使用被测对象名称
- `it` 使用 `should + 预期行为` 格式
- 覆盖正常路径和异常路径

---

## 测试覆盖要求

| 模块类型 | 覆盖率要求 | 重点测试 |
|----------|-----------|----------|
| 核心层 | >= 80% | 业务规则、不变量、状态流转 |
| 技术层 | >= 70% | 端口实现、错误处理、边界条件 |
| 策略层 | >= 60% | 路由逻辑、降级策略 |

---

## 测试原则

- Arrange-Act-Assert 模式组织测试代码
- 每个测试只验证一个行为
- 测试之间互不依赖，可独立运行
- 优先测试核心层业务规则，而非实现细节
- 禁止测试私有方法，通过公共接口间接验证

---

# 开发流程

## 新增功能流程

```text
1. 确认业务能力归属（参见 business-capabilities/）
2. 在核心层定义能力契约（接口）
3. 编写核心层业务逻辑 + 单元测试
4. 在技术层提供具体实现 + 集成测试
5. 在策略层组装依赖 + 验证流程
6. 更新知识库文档
```

---

## 代码审查清单

**代码质量**：
- [ ] 是否符合命名规范？
- [ ] 函数/文件是否超出长度限制？
- [ ] 是否存在魔法数字？
- [ ] 是否存在重复代码（三次法则）？

**架构合规**：
- [ ] 是否正确分层？
- [ ] 核心层是否依赖了技术实现？
- [ ] 模块间是否通过接口通信？
- [ ] 错误处理是否符合规范？

**测试覆盖**：
- [ ] 是否包含测试？
- [ ] 核心业务规则是否有单元测试？
- [ ] 是否覆盖异常路径？

**知识库同步**：
- [ ] 是否需要更新领域模型？
- [ ] 是否需要更新 API 契约？
- [ ] 是否需要更新术语表？
- [ ] 变更记录是否已填写（含关联变更）？
- [ ] 联动文档是否已检查（参见 knowledge-base/README.md 中的文档联动说明）？

## 提交规范

```text
[type]: [subject]

[body]
```

**type 类型**: feat / fix / docs / style / refactor / test / chore

**subject 规则**：
- 使用祈使句（如 `add` 而非 `added`）
- 不超过 50 字符
- 不加句号
