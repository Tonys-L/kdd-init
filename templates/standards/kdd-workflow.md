# 项目工作规则

本项目采用知识库驱动开发模式（Knowledge Driven Development）。

项目知识库是项目事实来源（Source of Truth）。

知识库聚焦代码无法表达的内容：约束、不变量、边界和术语。

所有分析、设计、开发、测试和文档更新工作必须基于知识库进行。

---

# 硬护栏（最高优先级，每次响应前必须执行）

## 1. 声明当前状态（每次响应必须输出）

```text
任务：[一句话描述]
门禁阶段：[知识库阅读 / 影响分析 / 设计方案 / 用户确认 / 开发实现 / 测试验证 / 知识库同步 / 轻量门禁]
```

## 2. 自检（未通过禁止输出代码）

```text
□ 是否已阅读相关知识库文档？
□ 是否已输出影响分析？
□ 是否已输出设计方案？
□ 是否已获得用户确认？
```

**规则**：自检未通过 → 只能输出分析、方案、提问，禁止输出代码。

**无例外**：无论任务大小、无论是否紧急，都必须通过自检才能输出代码。

**默认走完整门禁**：所有任务默认必须走完整门禁流程，不允许跳过任何阶段。

# 轻量门禁（仅限纯 UI 调整）

当且仅当同时满足以下全部条件时，可进入轻量门禁：

```text
□ 修改范围仅涉及 UI 样式/布局
□ 不修改业务逻辑/领域模型/接口契约
□ 不新增或删除业务模块文件（纯 UI 组件文件不在此限）
□ 不修改业务数据结构（纯 UI 展示属性变更不在此限，如样式类名、文案、可见性）
□ 不修改状态流转
```

### 判定示例

| 场景 | 判定 | 理由 |
|------|------|------|
| 修改按钮颜色、间距 | 轻量 | 仅涉及 UI 样式 |
| 调整页面布局 | 轻量 | 仅涉及 UI 布局 |
| 修改组件 props 的 `isLoading` 字段 | 轻量 | 纯 UI 展示属性变更 |
| 修改组件 props 新增业务数据字段 | **完整** | 修改了业务数据结构 |
| 新增表单字段且涉及后端校验 | **完整** | 修改了业务逻辑 |
| UI 组件新增状态管理（如 loading/error） | **完整** | 修改了状态流转 |
| 修改列表展示字段（字段来自后端新增） | **完整** | 修改了数据结构 |

轻量门禁只需输出：

```text
任务：[一句话描述]
门禁阶段：[轻量门禁]

【影响分析】

仅 UI 调整，不影响业务逻辑、数据结构、接口契约。

【修改范围】

- 涉及文件：xxx
- 修改内容：xxx

【知识库同步】

□ 是否需要更新经验教训库
□ 是否涉及组件接口变更
```

**任何一项不满足轻量条件，必须走完整门禁。**

**轻量门禁仍需用户确认后才能输出代码。输出后等待用户回复"确认"或"按此方案执行"方可进入开发。**

## 读代码

知识库记录代码无法表达的内容，代码能表达的内容必须从代码中获取。

阅读知识库后，必须阅读与任务相关的代码：

1. 阅读涉及修改的模块代码，理解当前实现
2. 阅读依赖和被依赖的模块代码，理解影响范围
3. 阅读相关测试代码，理解现有测试覆盖

禁止仅基于知识库就动手修改代码，必须同时理解代码现状。

### 代码定位方法

按以下顺序定位相关代码，避免遗漏：

```text
1. 修改目标：直接要修改的文件/函数

2. 调用链上游：谁调用了要修改的代码（被依赖方）
   - 搜索函数/方法名的引用
   - 搜索接口的实现类

3. 调用链下游：要修改的代码调用了谁（依赖方）
   - 查看当前代码的依赖引用

4. 测试代码：对应的测试文件
   - 查找同名测试文件（如 foo.ts → foo.test.ts）
   - 搜索引用被修改函数的测试

5. 配置代码：相关的配置/路由/注入
   - 搜索被修改模块的注册/配置位置
```

每一步都必须实际搜索，不得仅凭记忆或猜测跳过。

# 门禁二：影响分析

知识库阅读完成后进入：

```text

```

必须分析以下内容：

```text
□ 是否违反约束或不变量

□ 是否新增业务能力

□ 是否修改业务规则

□ 是否修改模块职责

□ 是否修改能力契约

□ 是否修改接口契约

□ 是否修改状态流转

□ 修改的代码是否被其他模块依赖（从代码中检查依赖和被依赖关系）

□ 是否需要新增/修改测试
```

---

## 状态机修改专项检查

当影响分析中"修改状态流转"为"是"时，必须额外执行：

```text
□ 列出所有受影响的状态转换路径（不仅是修改的那一条）
□ 检查每条路径是否仍然合法（对照 flows.md 中的转换规则和禁止转换）
□ 识别是否有间接影响：修改 A→B 是否导致 C→A 或 B→D 失效
□ 检查跨模块事件联动（对照 flows.md 跨模块事件联动表）：本次修改的事件是否触发其他模块的联动动作
```

---

## 必须输出

| 检查项 | 是否影响 | 说明 |
| ------ | ---- | -- |
| 违反约束/不变量 | 是/否 | 说明 |
| 新增业务能力 | 是/否 | 说明 |
| 修改业务规则 | 是/否 | 说明 |
| 修改模块职责 | 是/否 | 说明 |

| 修改接口契约 | 是/否 | 说明 |

| 新增/修改测试 | 是/否 | 说明 |

## 原则冲突裁决

设计原则详见 `{{KB_PATH}}/constraints.md`。

当多个原则发生冲突时，按以下优先级裁决：

```text
业务优先 > 职责优先 > 变更成本 > 简单优先 > 扩展优先
```

裁决示例：

- 满足当前需求 + 结构简单 + 扩展一般 **优于** 支持十种扩展 + 结构复杂 + 当前不需要
- 稍微增加设计成本 + 变更仅改 1 个模块 **优于** 开发快 + 变更需改 20 个模块

目标：**最小充分复杂度**

# 门禁五：开发实现

获得确认后进入：

```text
[开发实现]
```

开发过程中必须：

1. 遵循 constraints.md 中的架构约束
2. 不得违反业务不变量
3. 不得基于猜测补全业务规则
4. 不得脱离现有架构随意重构
5. 代码能表达的内容不需要写入知识库
6. 任何修改都必须有对应测试，包括受影响的已有功能
7. 不得超出"任务范围锁定"中"本次做"的范围
8. 发现额外问题时记录下来，不得顺手修改

# 知识库与代码不一致时

发现历史不一致（非当前开发引入）时：以代码实现为准，并立即更新相关知识库文档。

与"文档优先原则"的区别：
- 文档优先原则：开发过程中，先改文档再改代码（主动）
- 以代码为准：发现历史不一致时，以代码为真相修正文档（被动）

禁止长期存在代码与知识库不一致的情况。
