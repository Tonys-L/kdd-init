# 项目工作规则

本项目采用知识库驱动开发模式（Knowledge Driven Development）。

项目知识库是项目事实来源（Source of Truth）。

所有分析、设计、开发、测试和文档更新工作必须基于知识库进行。

---

# 硬护栏（最高优先级，每次响应前必须执行）

## 1. 声明当前状态（每次响应必须输出）

```text
任务：[一句话描述]
门禁阶段：[知识库阅读 / 影响分析 / 设计方案 / 用户确认 / 开发实现 / 测试验证 / 知识库同步 / 轻量门禁]
```

## 2. 自检（未通过禁止输出代码）

```text
□ 是否已阅读相关知识库文档？
□ 是否已输出影响分析？
□ 是否已输出设计方案？
□ 是否已获得用户确认？
```

**规则**：自检未通过 → 只能输出分析、方案、提问，禁止输出代码。

**无例外**：无论任务大小、无论是否紧急，都必须通过自检才能输出代码。

**默认走完整门禁**：所有任务默认必须走完整七步门禁流程，不允许跳过任何阶段。

**设计检查清单**（输出架构设计前必须检查）：

```text
□ 是否从业务能力出发？
□ 是否存在明确业务价值？
□ 是否职责清晰？
□ 是否存在隐式耦合？
□ 核心层是否依赖技术实现？
□ 是否存在不变量保护？
□ 是否存在过度设计？
□ 是否为未知变化设计？
□ 是否考虑变更成本？
□ 删除一个抽象后是否更简单？
□ 是否支持未来演进？
□ 是否便于测试？
```

如果存在不必要复杂度，应优先简化设计。

# 轻量门禁（仅限纯 UI 调整）

当且仅当同时满足以下全部条件时，可进入轻量门禁：

```text
□ 修改范围仅涉及 UI 样式/布局
□ 不修改业务逻辑/领域模型/接口契约
□ 不新增或删除业务模块文件（纯 UI 组件文件不在此限）
□ 不修改业务数据结构（纯 UI 展示属性变更不在此限，如样式类名、文案、可见性）
□ 不修改状态流转
```

### 判定示例

| 场景 | 判定 | 理由 |
|------|------|------|
| 修改按钮颜色、间距 | 轻量 | 仅涉及 UI 样式 |
| 调整页面布局 | 轻量 | 仅涉及 UI 布局 |
| 修改组件 props 的 `isLoading` 字段 | 轻量 | 纯 UI 展示属性变更 |
| 修改组件 props 新增业务数据字段 | **完整** | 修改了业务数据结构 |
| 新增表单字段且涉及后端校验 | **完整** | 修改了业务逻辑 |
| UI 组件新增状态管理（如 loading/error） | **完整** | 修改了状态流转 |
| 修改列表展示字段（字段来自后端新增） | **完整** | 修改了数据结构 |

轻量门禁只需输出：

```text
任务：[一句话描述]
门禁阶段：[轻量门禁]

【影响分析】

仅 UI 调整，不影响业务逻辑、数据结构、接口契约。

【修改范围】

- 涉及文件：xxx
- 修改内容：xxx

【知识库同步】

□ 是否需要更新经验教训库
□ 是否涉及组件接口变更
```

**任何一项不满足轻量条件，必须走完整门禁。**

**轻量门禁仍需用户确认后才能输出代码。输出后等待用户回复"确认"或"按此方案执行"方可进入开发。**

## 增量阅读

首次任务必须完整阅读必读文档。

后续任务可采用增量阅读：

1. 仅阅读与当前任务相关的文档（根据任务关键词匹配）
2. 如果上次任务修改了某些文档，本次任务涉及相同模块时必须重新阅读
3. 如果距离上次阅读超过 10 个任务，重新完整阅读必读文档

任务相关性判断：

| 任务类型 | 必须阅读 |
|----------|----------|
| 新增业务能力 | `business-capabilities/`、`domain-model/`、`architecture-overview/` |
| 修改数据结构 | `data-model/`、`domain-model/` |
| 修改 API | `api-contracts/`、`service-specs/` |
| 修改协议 | `protocol-docs/`、`service-specs/` |
| 修改架构 | `architecture-overview/`、`development-guide/` |
| 修复 Bug | `domain-model/`、对应业务能力的文档、`lessons/` |

---

## 按需读取

知识库已拆分为目录结构，每个目录有 `README.md`（TL;DR + 文件索引）+ 若干小文件。

**读取策略**：

1. 读取 `knowledge-base/README.md`，了解文档全貌
2. 读取相关目录的 `README.md`，通过 TL;DR 和文件索引定位需要的文件
3. **核心文档目录（project-context / business-capabilities / domain-model / architecture-overview / development-guide / data-model / glossary.md）：按文件索引定位，只读与任务相关的文件。`invariants.md` 必读**
4. **条件文档目录（service-specs / api-contracts / protocol-docs）：不涉及时仅读 README.md 的 TL;DR**
5. ⚠️ 标记的内容是红线约束，任何情况下都不可违反

**安全规则**：

- `domain-model/invariants.md` 是必读文件，任何任务都必须阅读
- README.md 中的 TL;DR 如果不够明确，必须展开阅读对应文件
- 条件文档仅在确认不涉及后才可跳过

示例：

```text
任务：新增用户注册功能

1. 读取 knowledge-base/README.md，定位相关目录

2. 读取相关目录的 README.md：
   - project-context/README.md → 了解项目约束
   - business-capabilities/README.md → TL;DR: 核心能力：用户管理、设备控制 ⚠️ 设备控制不包含设备管理
   - domain-model/README.md → TL;DR: 核心实体：User、Device ⚠️ 订单金额必须大于0

3. 按文件索引定位，读取相关文件：
   - domain-model/invariants.md（必读）
   - business-capabilities/capabilities.md（任务相关）
   - business-capabilities/boundaries.md（任务相关）
   - architecture-overview/dependencies.md（任务相关）

4. 条件文档，按需判断：
   - api-contracts/README.md → 涉及新增注册接口，读取 endpoints.md
   - service-specs/README.md → 不涉及状态机，仅保留 TL;DR
   - protocol-docs/README.md → 不涉及协议，仅保留 TL;DR
```

## 原则冲突裁决

设计原则详见 `knowledge-base/development-guide/design-principles.md`。

当多个原则发生冲突时，按以下优先级裁决：

```text
业务优先 > 职责优先 > 变更成本 > 简单优先 > 扩展优先
```

裁决示例：

- 满足当前需求 + 结构简单 + 扩展一般 **优于** 支持十种扩展 + 结构复杂 + 当前不需要
- 稍微增加设计成本 + 变更仅改 1 个模块 **优于** 开发快 + 变更需改 20 个模块

目标：**最小充分复杂度**

## 变更记录填写规范

每个被更新的文档必须填写变更记录，格式：

```text
| 日期 | 变更内容 | 变更人 | 关联变更 |
|------|----------|--------|----------|
| 2024-01-15 | 新增用户注册能力 | 张三 | #TASK-42 同步更新 domain-model/ |
```

**关联变更**字段填写规则：

- 同一批次的变更填写相同标记（如任务编号 `#TASK-42`）
- 联动更新的文档在此字段互相引用
- 初始化时填写 `—`

示例：

```text
# domain-model/ 变更记录
| 2024-01-15 | 新增 User 实体 | 张三 | #TASK-42 |

# data-model/ 变更记录
| 2024-01-15 | 新增 users 表 | 张三 | #TASK-42 同步 domain-model/ |
```

# 知识库与代码不一致时

发现历史不一致（非当前开发引入）时：以代码实现为准，并立即更新相关知识库文档。

与"文档优先原则"的区别：
- 文档优先原则：开发过程中，先改文档再改代码（主动）
- 以代码为准：发现历史不一致时，以代码为真相修正文档（被动）

禁止长期存在代码与知识库不一致的情况。
