# 项目工作规则

本项目采用知识库驱动开发模式（Knowledge Driven Development）。

项目知识库是项目事实来源（Source of Truth）。

所有分析、设计、开发、测试和文档更新工作必须基于知识库进行。

---

# 硬护栏（最高优先级，每次响应前必须执行）

## 1. 声明当前状态（每次响应必须输出）

```text
任务：[一句话描述]
门禁阶段：[知识库阅读 / 影响分析 / 设计方案 / 用户确认 / 开发实现 / 测试验证 / 知识库同步 / 轻量门禁]
```

## 2. 自检（未通过禁止输出代码）

```text
□ 是否已阅读相关知识库文档？
□ 是否已输出影响分析？
□ 是否已输出设计方案？
□ 是否已获得用户确认？
```

**规则**：自检未通过 → 只能输出分析、方案、提问，禁止输出代码。

**无例外**：无论任务大小、无论是否紧急，都必须通过自检才能输出代码。

**默认走完整门禁**：所有任务默认必须走完整七步门禁流程，不允许跳过任何阶段。

---

# 轻量门禁（仅限纯 UI 调整）

当且仅当同时满足以下全部条件时，可进入轻量门禁：

```text
□ 修改范围仅涉及 UI 样式/布局
□ 不修改业务逻辑/领域模型/接口契约
□ 不新增或删除业务模块文件（纯 UI 组件文件不在此限）
□ 不修改业务数据结构（组件 props 变更不在此限）
□ 不修改状态流转
```

轻量门禁只需输出：

```text
任务：[一句话描述]
门禁阶段：[轻量门禁]

【影响分析】

| 检查项    | 是否影响 | 说明 |
| ------ | ---- | -- |
| 新增业务能力 | 是/否  | 说明 |
| 修改业务规则 | 是/否  | 说明 |
| 修改领域模型 | 是/否  | 说明 |
| 修改模块职责 | 是/否  | 说明 |
| 修改能力契约 | 是/否  | 说明 |
| 修改接口契约 | 是/否  | 说明 |
| 修改协议定义 | 是/否  | 说明 |
| 修改配置模型 | 是/否  | 说明 |
| 修改数据结构 | 是/否  | 说明 |
| 修改状态流转 | 是/否  | 说明 |

【修改范围】

- 涉及文件：xxx
- 修改内容：xxx

【知识库同步】

□ 是否需要更新经验教训库
□ 是否涉及组件接口变更
```

**任何一项不满足轻量条件，必须走完整门禁。**

**轻量门禁仍需用户确认后才能输出代码。输出后等待用户回复"确认"或"按此方案执行"方可进入开发。**

# 必读文档

开始任何任务前，必须优先阅读以下文档：

项目知识地图：

```text
docs/knowledge-base/README.md
```

架构总览：

```text
docs/knowledge-base/architecture-overview.md
```

工程开发规范：

```text
docs/knowledge-base/development-guide.md
```

根据任务内容继续定位并阅读相关知识库文档。

---

# 任务状态机

所有任务必须按以下状态顺序执行：

```text
[知识库阅读]
        ↓
[影响分析]
        ↓
[设计方案]
        ↓
[用户确认]
        ↓
[开发实现]
        ↓
[测试验证]
        ↓
[知识库同步]
        ↓
[任务结束]
```

禁止跳过任何阶段。

未经用户确认：

禁止从设计阶段直接进入开发阶段。

---

# 门禁一：知识库阅读

开始任务后首先进入：

```text
[知识库阅读]
```

必须完成以下操作：

1. 阅读 docs/knowledge-base/README.md
2. 定位与任务相关的文档
3. 阅读相关文档
4. 阅读经验教训库（docs/knowledge-base/lessons/README.md），按业务分类匹配索引
5. 理解现有设计

---

## 必须输出

```text
【已阅读文档】

- xxx.md
- lessons/xxx.md

【现有设计理解】

- xxx
- xxx

【经验教训参考】

- 是否存在与当前任务相关的历史教训：是/否
- 相关教训摘要：xxx（若无则填"无"）
```

未输出上述内容，视为门禁未通过。

---

## 门禁未通过时

禁止输出：

* 代码
* 伪代码
* 文件修改方案
* 重构方案

只能继续知识库分析。

---

# 门禁二：影响分析

知识库阅读完成后进入：

```text
[影响分析]
```

必须分析以下内容：

```text
□ 是否新增业务能力

□ 是否修改业务规则

□ 是否修改领域模型

□ 是否修改模块职责

□ 是否修改能力契约

□ 是否修改接口契约

□ 是否修改协议定义

□ 是否修改配置模型

□ 是否修改数据结构

□ 是否修改状态流转
```

---

## 必须输出

| 检查项    | 是否影响 | 说明 |
| ------ | ---- | -- |
| 新增业务能力 | 是/否  | 说明 |
| 修改业务规则 | 是/否  | 说明 |
| 修改领域模型 | 是/否  | 说明 |
| 修改模块职责 | 是/否  | 说明 |
| 修改能力契约 | 是/否  | 说明 |
| 修改接口契约 | 是/否  | 说明 |
| 修改协议定义 | 是/否  | 说明 |
| 修改配置模型 | 是/否  | 说明 |
| 修改数据结构 | 是/否  | 说明 |
| 修改状态流转 | 是/否  | 说明 |

未输出分析结果，视为门禁未通过。

---

## 文档优先原则

如果分析结果显示：

```text
存在设计变更
```

则必须：

```text
先更新设计文档

后修改代码
```

禁止反向执行。

---

## 门禁未通过时

禁止输出：

* 代码
* 文件修改方案

只能继续影响分析。

---

# 门禁三：设计方案

影响分析完成后进入：

```text
[设计方案]
```

必须输出：

---

## 需求理解

说明：

* 当前需求是什么
* 需要解决什么问题

---

## 修改范围

说明：

* 涉及哪些模块
* 涉及哪些文档
* 涉及哪些契约

---

## 设计方案

说明：

* 如何实现
* 为什么这样实现
* 是否影响现有架构

复杂流程可使用 Mermaid。

---

## 风险分析

说明：

* 兼容性风险
* 性能风险
* 扩展性风险
* 维护风险

---

## 开发计划

说明：

* 实施步骤
* 验证方式
* 回滚方案

---

# 门禁四：用户确认

设计方案完成后：

必须等待用户确认。

---

未获得确认前：

禁止进入开发阶段。

禁止输出：

* 功能代码
* 文件修改内容
* 重构代码

---

# 门禁五：开发实现

获得确认后进入：

```text
[开发实现]
```

开发过程中必须：

1. 遵循架构原则
2. 遵循工程规范
3. 优先遵循知识库
4. 不得基于猜测补全业务规则
5. 不得脱离现有架构随意重构

---

# 门禁六：测试验证

代码完成后进入：

```text
[测试验证]
```

必须说明：

```text
已执行哪些测试

验证了哪些场景

是否通过
```

---

# 门禁七：知识库同步

开发完成后进入：

```text
[知识库同步]
```

必须检查：

```text
□ 是否需要更新业务文档

□ 是否需要更新架构文档

□ 是否需要更新契约文档

□ 是否需要更新接口文档

□ 是否需要更新经验教训库

□ 是否需要新增架构决策记录

□ 是否涉及组件接口变更？如果是，是否需要在经验教训库中记录？
```

---

## 必须输出

```text
【知识库更新结果】

需要更新：

- xxx.md
- xxx.md

更新内容：

- xxx
- xxx
```

或者：

```text
【知识库更新结果】

本次无需更新知识库。
```

---

# 知识库与代码不一致时

以代码实现为准。

并立即更新相关知识库文档。

禁止长期存在代码与知识库不一致的情况。

---

# 最终目标

```text
知识库驱动开发

设计先于编码

文档先于实现

测试保证质量

代码与知识同步

设计与实现一致
```
